### 1. Executive Summary

This document provides a complete system analysis for the Intent Solutions landing page (`intentsolutions.io`). The primary system is a static site generated by **Astro**, located in the `astro-site` directory. It uses **React** for interactive UI "islands" and **Tailwind CSS** for styling. The entire platform is deployed on **Netlify**, which builds and serves the site from the `astro-site` subdirectory of the main branch.

A key architectural finding is the presence of **two distinct web applications** in this repository. The deployed production system is the Astro site. A separate, root-level React/Vite application exists in `02-Src`, but it is **not the deployed application**. This legacy or alternative project appears to be a more traditional SPA, while the live site leverages Astro's static-first architecture for performance. This distinction is critical for all development and operational tasks.

The system's current state is live and stable on Netlify, serving traffic from the `main` branch. The architecture is designed for high performance, leveraging static generation, partial hydration, and a CDN. The operational workflow is straightforward: a `git push` to the `main` branch triggers an automated build and deployment on Netlify. All infrastructure is effectively managed by Netlify, with DNS handled by Porkbun.

This analysis focuses primarily on the operational **`astro-site` project**. The secondary React/Vite project is analyzed as a related but non-operational artifact. Understanding this dual-project structure is the most important initial takeaway for managing this system.

### 3. Directory Deep-Dive

#### `astro-site/` (🔑 PRIMARY PRODUCTION DIRECTORY)

This is the active, deployed project.

-   **`src/`**: Contains all source code.
    -   `pages/index.astro`: The main entry point and page layout. It assembles React components into a single page.
    -   `components/`: React components (`.tsx`) that are hydrated as interactive islands (e.g., `Contact.tsx`).
    -   `layouts/Layout.astro`: The base HTML shell, including SEO setup and global styles.
-   **`public/`**: Static assets like `favicon.svg` and `_redirects`.
-   **`tests/`**: Playwright E2E tests.
-   **`astro.config.mjs`**: Astro configuration file, defining integrations (React) and the site URL.
-   **`package.json`**: Defines all dependencies and scripts (`dev`, `build`, `test`).
-   **`netlify.toml`**: **This file is at the root**, but its `base = "astro-site"` directive makes it control this directory's deployment.

#### `02-Src/` (Legacy/Alternative Project)

This directory contains a standard React/Vite SPA. It is **NOT** what is deployed.

-   **`App.tsx`**: Main application component with React Router setup.
-   **`pages/Index.tsx`**: The main landing page component for this version.
-   **`components/`**: A different set of React components, structured similarly to the Astro site but for a full SPA.
-   **`vite.config.ts`**: Build configuration for this SPA.
-   **Gaps**: The purpose and operational status of this application are undocumented. It's unclear if it's a template, a previous version, or an abandoned effort. This is a source of potential confusion.

#### `01-Docs/`

-   **Key Documents**: Contains a wealth of project information.
    -   `023-ref-project-overview.md`: A good high-level summary, though it seems to describe the legacy React app more than the Astro one.
    -   `026-dpl-netlify-deployment-guide.md`: **Crucial operational guide**. Accurately describes the Netlify/Porkbun setup.
    -   `030-des-architecture-documentation.md`: Describes the architecture of the legacy React app.
-   **Gaps**: The documentation is not consolidated and contains conflicting information due to the dual-project structure. It's not always clear which document applies to which project.

#### `tests/` (Root)

-   This directory is largely empty, containing only a `README.md` and some empty subdirectories. The actual, working tests are located in `astro-site/tests/`.

#### `netlify.toml` (Root) 🔑

**CRITICAL FILE**. This file at the root of the repository controls the entire deployment process.

-   `base = "astro-site"`: This is the most important line. It tells Netlify to run all commands from within the `astro-site` directory.
-   `command = "bun install && bun run build"`: The build command executed by Netlify.
-   `publish = "dist"`: The directory (relative to `base`) that Netlify deploys.
-   `redirects`: Defines HTTP->HTTPS, www->non-www, and security headers.

### 4. Operational Reference

#### Deployment Workflows

##### Local Development (`astro-site`)
1.  **Required tools**: `bun` (v1.0.0+), `node` (v20+)
2.  **Environment setup**:
    ```bash
    cd astro-site
    bun install
    ```
3.  **Running locally**:
    ```bash
    bun run dev
    ```
    The site will be available at `http://localhost:4321`.

##### Staging Deployment
There is no formal staging environment. Changes are deployed directly to production from the `main` branch. A preview deployment can be manually created from a feature branch in the Netlify UI if needed.

##### Production Deployment
-   **Trigger**: Automated on `git push` to the `main` branch.
-   **Pre-deployment checklist**:
    -   [ ] All E2E tests in `astro-site` are passing locally (`bun run test`).
    -   [ ] Changes have been reviewed and approved via Pull Request.
    -   [ ] Rollback plan: Revert the merge commit or deploy a previous commit from the Netlify UI.
-   **Deployment command**: `git push origin main`
-   **Monitoring during deployment**: Watch the "Deploys" section in the Netlify UI for the build status.
-   **Rollback command**: In Netlify UI: Go to "Deploys", select a previous successful deploy, and click "Publish deploy".

### 7. Development Workflow

#### CI/CD Pipeline
-   **Tool**: Netlify CI/CD.
-   **Trigger**: `git push` to `main`.
-   **Pipeline Definition**: The entire pipeline is defined in `netlify.toml`.
    1.  Netlify detects push.
    2.  Clones the repository.
    3.  Changes directory to `astro-site`.
    4.  Runs `bun install && bun run build`.
    5.  Deploys the resulting `dist/` directory to the CDN.
-   **Gaps**: The pipeline does not run tests. Tests are a manual, local-only step.
